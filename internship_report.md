## Постановка задачи

### 1. Актуальность работы
В современном цифровом мире мобильные приложения, в частности для iOS, стали неотъемлемой частью повседневной жизни миллионов пользователей. Они обрабатывают колоссальные объемы конфиденциальной информации: от персональных идентификаторов и финансовых данных до геолокации, здоровья и биометрических показателей. Повышенное внимание к конфиденциальности данных со стороны регуляторов (таких как GDPR в Европе, CCPA в Калифорнии и собственные правила App Store) и самих пользователей делает безопасность обработки этих данных критически важной [1].

Несмотря на это, множество приложений по-прежнему содержат уязвимости и используют небезопасные практики программирования, приводящие к утечкам данных. По данным отчета за 2023 год, более 40% мобильных приложений содержат как минимум одну уязвимость высокой степени риска, связанную с неправильной обработкой данных [2]. Проблема усугубляется тем, что традиционные процессы ревью кода в условиях сжатых сроков выхода продукта на рынок (time-to-market) часто не способны выявить все потенциально опасные паттерны кода.

Таким образом, существует острая потребность в автоматизированных инструментах, способных масштабируемо, точно и эффективно анализировать исходный код iOS-приложений на предмет наличия небезопасных практик работы с пользовательскими данными.

### 2. Формулировка проблемы
Существующие инструменты статического анализа безопасности приложений (Static Application Security Testing, SAST) зачастую:
*   Ориентированы на общие уязвимости и не покрывают весь спектр iOS-специфичных практик, связанных с конфиденциальностью [3].
*   Используют шаблонные подходы (например, поиск по сигнатурам), которые не всегда способны выявить сложные, контекстно-зависимые сценарии утечек [4].
*   Требуют значительной ручной настройки и экспертных знаний для поддержания актуальности набора правил, что затрудняет их использование в быстро меняющейся экосистеме iOS.

Кроме того, с появлением больших языковых моделей (LLM) открывается новая возможность для семантического анализа кода, однако их применимость и эффективность для решения конкретной задачи обнаружения практик работы с данными в iOS требуют отдельного экспериментального исследования [5].

## Обзор предметной области

Для решения сформулированной проблемы применяются различные методы и подходы, как классические, так и инновационные.

### 1. Статический анализ на основе абстрактных синтаксических деревьев (AST) и графов потока данных (DFG)
*   **Описание:** Исходный код (Swift/Objective-C) парсится в Абстрактное Синтаксическое Дерево (AST). На следующем этапе строится Граф Потока Управления (CFG) и Граф Потока Данных (DFG), отслеживающий перемещение значений переменных.
*   **Анализ применимости:** Высокая точность за счет понимания контекста. Инструменты вроде SourceKit предоставляют готовые средства для парсинга. Трудности заключаются в анализе ООП кода с полиморфизмом, работе с рефлексией и возможном большом количестве ложных срабатываний.

### 2. Анализ на основе шаблонов и сигнатур (Pattern Matching / Grep-based)
*   **Описание:** Поиск в коде предопределенных шаблонов, строк или сигнатур вызовов API (например, `UserDefaults.standard.set` для паролей).
*   **Анализ применимости:** Главные преимущества – скорость и простота реализации (используется в MobSF [7]). Ключевой недостаток – низкая контекстная осведомленность, что приводит к ложным срабатываниям и неспособности обнаружить сложные многошаговые утечки.

### 3. Анализ зависимостей (Taint Analysis)
*   **Описание:** Специализированный вид статического анализа. Данные из доверенных источников (sources) помечаются как "загрязненные". Анализатор отслеживает распространение "загрязнения". Если данные достигают критического стока (sink) без санитайзера, фиксируется уязвимость.
*   **Анализ применимости:** Считается "золотым стандартом" (реализовано в CodeQL [8]). Требует значительных вычислительных ресурсов и экспертизы для написания качественных правил.

### 4. Анализ с использованием больших языковых моделей (LLM)
*   **Описание:** Использование LLM (GPT, CodeBERT, CodeLlama) для семантического анализа кода на естественном языке.
*   **Анализ применимости:** Преимущество – способность понимать абстрактные концепции и снижать количество ложных срабатываний. Риски: высокая стоимость вычислений, "галлюцинации", зависимость от промптов. На текущий момент чаще используются как вспомогательный инструмент.

## Альтернативные решения

Для задачи автоматического выявления небезопасных практик обработки пользовательских данных в iOS-приложениях существует ряд готовых инструментов, однако ни одно из них не обеспечивает полного покрытия поставленной задачи.

1.  **MobSF (Mobile Security Framework) [7]:** Открытая платформа. *Недостаток:* ограниченный анализ потоков данных, опора на сигнатурные правила.
2.  **Коммерческие SAST-платформы (Checkmarx, Fortify, Veracode) [9]:** Высокий уровень зрелости. *Недостаток:* ориентация на кроссплатформенную безопасность, недостаточная точность для специфичных iOS сценариев, высокая стоимость.
3.  **Xcode Static Analyzer и SwiftLint [10]:** Стандартные инструменты. *Недостаток:* не предназначены для анализа безопасности пользовательских данных, не поддерживают модели источников и стоков.
4.  **GitHub Advanced Security / CodeQL [8]:** Семантический анализ. *Недостаток:* текущие наборы запросов для iOS не покрывают большинство сценариев работы с пользовательскими данными «из коробки».

### Сравнительная таблица решений

| Решение | Тип анализа | Поддержка Swift | Анализ потоков данных (Taint) | Ориентация на пользовательские данные | Готовность к использованию |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **MobSF** | Статический + Динамический | Частичная | Ограниченная (шаблонный) | Частичная | Да |
| **Checkmarx / Fortify** | Статический (SAST) | Да | Частичная | Частичная | Да |
| **Xcode Static Analyzer** | Статический | Да | Нет | Нет | Да |
| **SwiftLint** | Статический (линтер) | Да | Нет | Нет | Да |
| **GitHub CodeQL** | Статический (семантический) | Частичная | Нет | Ограниченная | Да |

**Вывод:** Анализ показал отсутствие инструмента, обеспечивающего полноценное автоматическое обнаружение небезопасных практик работы с пользовательскими данными в iOS-приложениях «из коробки». В связи с этим предлагается разработка собственного программного механизма на основе гибридного анализа (шаблоны + упрощенный Taint Analysis) с перспективой интеграции LLM.

## Функциональные возможности

### Архитектура решения
Решение построено по модульному принципу и состоит из трех основных компонентов:

1.  **Модуль статического анализа:** Ядро системы на базе **SwiftLint**. Использует кастомные правила для выявления нарушений безопасности.
2.  **LLM-компонент:** "Виртуальный эксперт" для верификации сложных случаев через внешнее API (GPT-4, CodeLlama).
3.  **Модуль сравнения и оценки:** Консолидация результатов, расчет метрик (Precision, Recall) и генерация отчетов.

### Входные данные
1.  **Исходный код iOS-приложения:** Файлы `.swift`, структура проекта, учет комментариев.
2.  **Конфигурационный файл (YAML/JSON):**
    *   `ruleset`: Список активных правил безопасности.
    *   `llm_integration`: Настройки подключения к LLM.
    *   `sources_sinks`: Настраиваемые списки источников и стоков.
    *   `ignored_paths`: Исключаемые директории (Pods, Carthage).

### Механизмы обработки данных (Алгоритм)
1.  **Парсинг и первичный анализ:** Рекурсивный обход файлов, запуск SwiftLint с кастомными правилами (regex/AST-паттерны).
2.  **Верификация с помощью LLM:** Найденные нарушения отправляются в LLM с контекстом. LLM возвращает JSON с оценкой вероятности уязвимости (`is_vulnerable`, `confidence`). Ложные срабатывания отфильтровываются.
3.  **Сравнение и оценка:** Сравнение результатов с эталонным инструментом (MobSF) для расчета метрик качества.

### Выходные данные
*   **Отчет об уязвимостях (JSON/HTML):** Детальная информация о найденных проблемах (файл, строка, серьезность, сниппет кода, источник обнаружения).
*   **Сравнительная таблица (CSV):** Результаты верификации (True Positive, False Positive и т.д.).

### Обоснование набора критериев безопасности
Набор правил основан на стандарте **OWASP MASVS (V2: Data Storage and Privacy Requirements)** и охватывает 10 критичных сценариев:
1.  **MSTG-STORAGE-1:** Хранение паролей/PIN в UserDefaults.
2.  **MSTG-STORAGE-2:** Хранение токенов/API ключей в plain text.
3.  **MSTG-STORAGE-3:** Вывод чувствительных данных в логи (print, NSLog).
4.  **MSTG-STORAGE-6:** Отсутствие шифрования перед сохранением в ФС.
5.  **MSTG-PRIVACY-1:** Фиксация геолокации без согласия.
6.  **MSTG-STORAGE-14:** Недостаточный уровень защиты Keychain.
7.  **MSTG-STORAGE-15:** Небезопасное кэширование веб-контента.
8.  **MSTG-STORAGE-16:** Небезопасное хранение в Core Data.
9.  **MSTG-STORAGE-4:** Кэширование клавиатурного ввода.
10. **MSTG-STORAGE-5:** Отсутствие очистки конфиденциальных данных из памяти.
